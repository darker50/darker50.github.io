    • 简介

    去年做自动的时候遇到修改过这个开源的框架，也自己修改过，当时也为这个问题纠结了好久，主要开源代码的外面是ant工程，里面是类maven工程，自己傻傻的就认为直接从github下载下来就可以直接修改了，结果查了两天，还是放弃了

    在github，搜索下就reportNG，就可以直接找到这个开源的框架，感觉还是会有懒人存在，地址给出
    ReportNG

    开工

    下载完成解压，出来的看到的就是一个build.xml，这个都知道是Ant的工程文件，但这个不是我们用的，进入reportng目录，这回看到有pom.xml文件和两个目录，其中lib目录就是引用包，reportng是使用Velocity模板引擎把测试结果写到报告中，src就是我们源码的目录

    maven工程结构

    目录 说明
    src/main/resources Application/Library resources
    src/main/resources-filtered Application/Library resources which are filtered. (Starting with Maven 3.4.0, not yet released.)
    src/main/filters Resource filter files
    src/main/webapp Web application sources
    src/test/java Test sources
    src/test/resources Test resources
    src/test/resources-filtered Test resources which are filtered by default. (Starting with Maven 3.4.0, not yet released.)
    src/test/filters Test resource filter files
    src/it Integration Tests (primarily for plugins)
    src/assembly Assembly descriptors
    src/site Site
    LICENSE.txt Project’s license
    NOTICE.txt Notices and attributions required by libraries that the project depends on
    README.txt Project’s readme

    这个目录结构里面还少一个配置文件，maven使用的是仓库管理jar的方式，配置信息都写在pom.xml文件里面，还有别的目录，这里不细说了，自己去了解用处吧。

    细心的对比下，就知道reportNG的源码和maven目录结构有点不一样，动手吧，直接把reportNG的目录下的文件都放到标准的maven的目录结构中，这个时候，一个正常的开源代码环境就整理好了，下面开始修改代码了。

    进阶

    老打成jar包也是麻烦啊，目录不是正好有单元测试的代码吗？结果跑完的结果显示的是testNG的报告，它只是测试方法的可用性而已，如果要实时调试模板的方法就是把源码集成到项目中，简单的说是把原来导入jar的方法删除，再把源码按修改完成的maven目录放到自己的项目就可以了。这个时候想怎么修改就怎么修改吧。
    最后，使用maven的一定要了解maven目录结构各个目录的说明啊，真的很重要，很重要，很重要
